1) DESIGN PATTERNS
------------------

1.1 Strategy Pattern (Filtering)
Purpose:
  You apply different “filter rules” (strategies) to lists of Venues and VenueBookings
  without hardcoding every condition inside the controller.

Where it is implemented:

A) Venue filtering strategies
  - Strategy interface:
    src\main\java\com\faculty\ems\strategy\VenueFilterStrategy.java

  - Concrete strategy implementations:
    src\main\java\com\faculty\ems\strategy\impl\SearchFilterStrategy.java
    src\main\java\com\faculty\ems\strategy\impl\CapacityFilterStrategy.java
    src\main\java\com\faculty\ems\strategy\impl\StatusFilterStrategy.java

  - Strategy “context” (the class that applies all strategies):
    src\main\java\com\faculty\ems\service\VenueFilterService.java

  - Where it is used (controller calls context service):
    src\main\java\com\faculty\ems\controller\VenueController.java
      - listVenues(...) builds VenueFilterCriteria and calls venueFilterService.filter(...)

B) Booking filtering strategies
  - Strategy interface:
    src\main\java\com\faculty\ems\strategy\BookingFilterStrategy.java

  - Concrete strategy implementations:
    src\main\java\com\faculty\ems\strategy\impl\VenueBookingFilter.java
    src\main\java\com\faculty\ems\strategy\impl\SocietyBookingFilter.java
    src\main\java\com\faculty\ems\strategy\impl\StatusBookingFilter.java

  - Strategy “context”:
    src\main\java\com\faculty\ems\service\BookingFilterService.java

  - Where it is used:
    src\main\java\com\faculty\ems\controller\VenueController.java
      - viewBookingHistory(...) builds BookingFilterCriteria and calls bookingFilterService.filter(...)

Why it is “Strategy”:
  - You have an interface (Strategy) + multiple implementations (Concrete Strategies)
  - The context collects them and applies them
  - New rules can be added by creating a new strategy class without rewriting the controller


1.2 Builder Pattern (Lombok @Builder)
Purpose:
  Create objects cleanly without huge constructors and many setters.

Where it is applied:
  - Entities/objects using Lombok builder:
    src\main\java\com\faculty\ems\model\Society.java              (@Builder)
    src\main\java\com\faculty\ems\model\User.java                (User.builder() is used in service)
    src\main\java\com\faculty\ems\model\SocietyAdminRequest.java (created with SocietyAdminRequest.builder())

  - Criteria DTOs using builder:
    src\main\java\com\faculty\ems\dto\VenueFilterCriteria.java    (VenueFilterCriteria.builder())
    src\main\java\com\faculty\ems\dto\BookingFilterCriteria.java  (BookingFilterCriteria.builder())

Examples of usage location:
  - VenueController builds filtering criteria:
    src\main\java\com\faculty\ems\controller\VenueController.java

  - Society admin request creation uses builder:
    src\main\java\com\faculty\ems\service\SocietyAdminRequestService.java

Where it is applied:

    @Service:     src\main\java\com\faculty\ems\service\*.java
    @Repository:  src\main\java\com\faculty\ems\repository\*.java

2) SERIALIZATION
----------------

Important note:
  In this project, “serialization” mainly appears as:
  - HTTP responses being converted to JSON automatically by Spring (Jackson)
  - DTOs/forms moving data between HTML forms and controllers (data binding)

2.1 Spring MVC JSON Serialization (@ResponseBody)
Where:
  src\main\java\com\faculty\ems\controller\CalendarController.java

What happens:
  - Endpoint: GET /calendar/monthly-bookings
  - Method returns: Map<String, Object>
  - Because of @ResponseBody, Spring converts the returned Map to JSON.
  - Browser (frontend JS) can call this endpoint and receive JSON.


2.2 Server-side form binding (DTO / ModelAttribute)
Where:
  - Venue booking form uses DTO:
    src\main\java\com\faculty\ems\dto\VenueBookingFormDto.java
    src\main\java\com\faculty\ems\controller\BookingController.java
      - @ModelAttribute("booking") VenueBookingFormDto bookingForm

  - Society admin request form uses DTO:
    src\main\java\com\faculty\ems\dto\SocietyAdminRequestDto.java
    src\main\java\com\faculty\ems\controller\UserManagementController.java
      - @ModelAttribute("request") SocietyAdminRequestDto dto

This is not Java “Serializable”, but it is still a core “serialization/data binding” concept:
  HTML form fields -> DTO fields -> server processing -> view rendering/redirect


3) HIBERNATE / JPA (ORM)
------------------------

3.1 Entity mapping (Hibernate maps Java objects to DB tables)
Where:
  Entities are in:
    src\main\java\com\faculty\ems\model\

Common annotations used:
  - @Entity, @Table
  - @Id, @GeneratedValue
  - @ManyToOne, @JoinColumn (relationships)
  - Hibernate timestamp annotations like @CreationTimestamp

Examples:
  - Venue:
    src\main\java\com\faculty\ems\model\Venue.java
      - @Entity @Table(name="venues")
      - unique venue name is enforced at DB level using:
        @Column(nullable=false, unique=true)

  - Event:
    src\main\java\com\faculty\ems\model\Event.java
      - @ManyToOne society
      - @ManyToOne organiser

  - VenueBooking:
    src\main\java\com\faculty\ems\model\VenueBooking.java
      - @ManyToOne event / venue / society / requestedBy / reviewedBy

  - Society / SocietyMember / SocietyAdminRequest:
    src\main\java\com\faculty\ems\model\Society.java
    src\main\java\com\faculty\ems\model\SocietyMember.java
    src\main\java\com\faculty\ems\model\SocietyAdminRequest.java
